## نموذج الأمان في روش

عشان تقيّم أي بلاغ صح، لازم تفهم إزاي المشروع متبني:

### الـ anon key معلن عن قصد — ومش ثغرة

المفتاح اللي في [`js/config.js`](js/config.js) هو **publishable anon key** بتاع Supabase، ومصمم إنه يبقى في كود المتصفح. أي حد يقدر يشوفه، وده طبيعي.

**الحماية الحقيقية في Row Level Security (RLS)** على مستوى قاعدة البيانات. المفتاح لوحده مايسمحش بأي حاجة أكتر من اللي السياسات مسموح بيها.

> لو لقيت طريقة تعمل بيها حاجة **مش مسموح بيها في السياسات** — دي ثغرة حقيقية وبلّغ عنها فورًا.

### الضمانات الأساسية

| الضمانة | إزاي متطبقة |
| --- | --- |
| مفيش حد يكتب باسم حد تاني | كل سياسات الكتابة مربوطة بـ `auth.uid()` |
| الرسايل الخاصة خاصة فعلًا | `conversations` و `messages` القراءة فيها للأطراف بس |
| المحفوظات والأصوات سرية | `bookmarks`, `poll_votes`, `post_views` قراءة لصاحبها بس |
| مفيش تلاعب في العدادات | العدادات بتتحدث من Triggers جوه Postgres، والكلاينت مالوش صلاحية `UPDATE` على `posts` |
| مفيش تصويت مزدوج | مفتاح أساسي على `(post_id, user_id)` + الدالة `poll_vote` بتقفل الصف |
| الملفات مش بتتلخبط | Storage RLS بتحصر الرفع في فولدر `auth.uid()` بتاع المستخدم |
| الدوال الداخلية مش متاحة | `REVOKE EXECUTE` على كل دوال الـ Triggers من `anon` و `authenticated` |
| مفيش XSS | كل نصوص المستخدم بتعدّي على `esc()` أو `linkify()` قبل ما تتحط كـ HTML |

### حاجات معروفة (مش ثغرات)

- **مفيش rate limiting على مستوى التطبيق** — بنعتمد على حدود Supabase الافتراضية. لو المشروع كبر، ده أول حاجة تتضاف.
- **مفيش فلترة محتوى تلقائية** — المنصة مافيهاش moderation أو حظر مستخدمين لسه.
- **البريد مش مؤكد افتراضيًا** — لو فعّلت "Confirm email" في Supabase هيبقى أأمن.
